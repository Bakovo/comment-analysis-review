# 质量检查

最终输出主题、观点表、优先级排序或业务动作前，先用这份清单检查。

## 范围检查

- 写清输入量、最终量、来源边界和时间范围。
- 区分原始评论条数和抽取后的观点行数。
- 不同组别比较时，必须使用一致的采集窗口、筛选规则和来源规则。
- 说明指标基于评论条数、观点行数、用户数、订单数，还是其他单位。

## 流程合规检查

- 确认 7 个流程步骤都按顺序覆盖。
- 如果某一步无法完成，说明限制，不要直接跳过。
- 如果最终交付跳过流程，或把方法用在不对应的步骤里，不要通过质检。
- 确认方法映射明确：粗探索使用词频/关键词、TF-IDF、共现分析；主题发现使用 LDA、NMF、Embedding/BERTopic、聚类或人工归类；观点抽取使用方面级情感分析或大模型结构化抽取。
- 确认整体情感分析只是趋势辅助，不能替代方面级观点行。

## 标签体系检查

- 分析方面标签应能跨评论复用。
- 合并含义重复的标签。
- 拆开把产品特征、用户情绪和业务动作混在一起的标签。
- 合并标签时保留别名说明。
- 做优先级排序前，先稳定标签体系。

## 标签检查

- 一行只表达一个观点。
- 分析方面命名目标方面，不要复述整句话。
- 观点概括说明用户对该方面的看法。
- 情绪必须和证据片段一致。
- 原文证据应尽量是原始评论中的直接片段。
- 业务动作要具体，并且和该观点有关。
- 当证据来自推断、含糊表达、宽松翻译或原文之外的信息时，降低置信度。

## 方法检查

- 词频、TF-IDF、共现结果只作为探索线索。
- 模型生成主题必须有代表评论支撑。
- 会影响业务动作的结论，优先使用方面级观点抽取。
- 高频和高优先级标签要抽样复核。
- 检查正面、负面、中性观点是否使用同样严格的编码标准。

## 抽样质检

最终交付前：

- 小数据集至少检查 20 行。
- 大数据集在可行时检查 5-10% 或至少 50 行。
- 对低频但严重的问题增加抽样。
- 对长评论、混合情绪、讽刺表达或翻译评论增加抽样。
- 记录反复出现的标签分歧，并更新标签体系。

## 优先级检查

- 分开看提及量、负面量、负面率、互动量、严重性和优先级分数。
- 说明“最高”指的是声量最高、负面率最高、互动最高、严重性最高还是业务风险最高。
- 不要用极小样本做强排序。
- 不要让低声量的高负面率问题在没有解释的情况下超过重复出现的严重问题。
- 证据有限时要写置信度或 caveat。

## 洞察检查

可用于决策的洞察应包含：

- 发现
- 范围
- 指标
- 原文证据
- 解释
- 业务动作
- 置信度
- 下一步验证

## 红旗

出现以下情况时，暂停并修改：

- 主题只来自关键词，没有代表评论
- 大多数评论都只生成一行，尽管原文有混合内容
- 大多数分析方面标签只出现一次
- 原文证据是改写，不是原文片段
- 业务动作很泛，比如“优化产品”
- 结论在比较样本量差异很大的分群
- 建议没有考虑严重性、可行性或决策成本
