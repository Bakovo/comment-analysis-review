# 分析师 Playbook

当任务需要专业级评论分析，而不只是快速摘要时，使用这份 playbook。

## 来源基础

这个 skill 基于：

- 用户提供的《评论分析流程》PPT，尤其是 7 步流程和方法选择表
- 常见用户反馈分析实践：数据清洗、编码、主题发现、方面级情感、证据抽样、洞察优先级判断
- 常见文本分析方法：词频、TF-IDF、共现分析、LDA、NMF、Embedding、BERTopic 和大模型结构化抽取

除非后续单独加入外部来源，否则不要把这些内容表述成学术引用。

## 标签体系设计

正式计数前，先建立稳定的 aspect 标签体系。

好的 aspect 标签应：

- 命名稳定的目标方面，如 `battery_life`、`pricing`、`delivery`、`comfort`、`setup`、`customer_service`
- 足够宽，能归纳重复反馈
- 足够窄，能支持具体动作
- 不把产品属性、用户情绪和业务动作混在一个标签里

避免：

- 每条评论都新建一次性标签
- 用整句话当标签
- 把原因和结果混在一起，如 `battery_bad_need_refund`
- 并行保留含义重叠的标签，如 `overheating`、`phone_heat`、`thermal_issue`

标签重叠时，合并标签并保留别名说明。

## 编码规则

一行只表达一个观点。

需要拆分评论的情况：

- 提到多个方面
- 同时出现表扬和抱怨
- 同一方面同时出现原因和结果
- 同一条评论里同时有服务问题和产品问题

不需要拆分的情况：

- 两个短语重复表达同一观点
- 某个短语只是背景信息
- 后半句只是加强语气，不是新方面

`evidence` 必须尽量是原文直接片段。如果含义来自推断，降低 `confidence`。

## 情绪规则

情绪应落在 aspect 层级。

- `positive`：明确表扬、满意、惊喜、推荐或超出预期
- `negative`：抱怨、摩擦、故障、预期落差、退款意图或流失信号
- `neutral`：事实性提及，没有明显评价
- `mixed`：同一个方面上正负情绪无法拆开

如果一条评论对不同方面有不同情绪，优先拆成多行，而不是整体标为 `mixed`。

## 抽样质检

专业交付前要抽样复核标签：

- 小数据集至少检查 20 行。
- 大数据集检查 5-10% 或至少 50 行，按实际可行性选择。
- 对高优先级方面、低频高风险问题、模型生成标签增加抽样。
- 记录反复出现的编码分歧，并更新标签体系。

如果观点表由大模型生成，要重点检查遗漏观点、编造证据、标签过细和情绪漂移。

## 分群分析

只有当分群能回答业务问题时才做分群。

常见分群：

- 平台或渠道
- 产品版本
- 评分区间
- 客户类型
- 地区或语言
- 时间周期
- 获客来源

比较分群前，先检查样本量、采集规则和时间窗口是否可比。

## 优先级逻辑

不要只用一个指标排序问题。

综合考虑：

- 声量：出现频率
- 负面率：问题集中程度
- 严重性：是否涉及伤害、流失、退款或安全
- 互动量：是否引发公开关注
- 可修复性：团队是否能行动
- 战略价值：是否影响定位、留存或转化

默认解释：

```text
高优先级 = 重复出现 + 负面 + 严重或可行动
中优先级 = 重复出现但严重性低，或严重但证据少
低优先级 = 孤立、含糊或不可行动
```

## 洞察写法

强洞察应包含：

- 用户在说什么
- 涉及哪个方面
- 有多常见或多重要
- 代表性证据
- 为什么重要
- 业务下一步做什么
- 置信度和 caveat

弱洞察：

```text
用户不喜欢这个产品。
```

强洞察：

```text
散热管理是一个早期产品风险信号：5 条样例评论中有 1 条提到玩游戏十分钟后手机发烫，并且这条负面观点的互动量最高。由于样本很小，应先作为假设处理，下一步查看客服工单，并拉取更大样本专门检查游戏发热相关评论。
```

## 常见失败模式

- 没说清楚就把评论条数和观点行数混用
- 只看关键词，不读代表评论，就直接命名主题
- 用极小样本说成趋势
- 混淆负面率、负面量和优先级
- 标签过细，无法复用
- 翻译情绪表达时过于字面，丢失真实意图
- 业务建议没有回到证据
